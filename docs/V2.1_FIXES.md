# v2.1 Patch Notes (post first-run debugging)

## Symptom that triggered v2.1

User ran v2.0 end-to-end and observed:
- `script 18` produced empty `unity_cards.json`
- All downstream stages cascaded to empty
- `script 13` rejection log showed **94% truncation rejection** (939 of 1000 generated posts)
- Of the 6% that survived, every card got `verdict=UNCERTAIN` because `p_fake` defaulted to 0.5

## Root causes diagnosed

### Bug 1: Schema mismatch between script 12 output and v2.0 script 13/18 expectations

**What v2.0 expected:**
```python
record = {
  "text": "...",
  "detectors": {
    "p_roberta_fake": 0.91,
    "p_distil_fake": 0.84,
    "p_degnn_fake": 0.78,
    "p_ensemble_fake": 0.84,
  }
}
```

**What script 12 actually emits (verified by reading the source):**
```python
record = {
  "text": "...",
  "p_roberta_fake": 0.91,
  "p_distil_fake": 0.84,
  "p_degnn_fake": 0.78,
  "r_pca_0": 0.42, "d_pca_0": -0.31, ...
  # NO "detectors" dict, NO "p_ensemble_fake"
}
```

Script 13 v2.0 looked up `rec.get("detectors", {}).get("p_ensemble_fake")` → always `None` → `p_fake=0.5` → all UNCERTAIN.

### Bug 2: Truncation gate too aggressive on real GPT-2 output

The legacy GPT-2 generator runs with `--max_new_tokens 120`, which means most generations are clipped mid-sentence (no terminal `.`/`!`/`?`). My v2.0 truncation gate rejected any text ending without terminal punctuation.

Result: 94% of valid GPT-2 generations were thrown away. The remaining 6% just happened to end at `.` by accident.

### Bug 3: Schema validator duplicated the gate

`UnityCard.text` had its own `_validate_text_complete` validator that re-applied the same terminal-punctuation rule. Even if script 13 let a card through, the schema would reject it again at script 18.

## Fixes in v2.1

### Fix 1: Script 13 reads correct schema

```python
# v2.1 — looks at top-level fields where script 12 actually puts them
p_roberta = rec.get("p_roberta_fake")           # was rec["detectors"]["p_roberta_fake"]
p_distil  = rec.get("p_distil_fake")
p_degnn   = rec.get("p_degnn_fake")
p_ensemble = mean([p for p in (p_roberta, p_distil, p_degnn) if p is not None])
```

Also computes `p_ensemble_fake` if not present (averages available detector probabilities) so script 18 has a sane verdict signal.

### Fix 2: Lenient truncation gate

`is_truncated()` now only flags genuinely broken fragments:
- Empty string
- < 30 characters  
- Ends in a Tagalog/English function word (`ang`, `ng`, `the`, `for`, etc.)

Text without terminal punctuation but ending on a content word now passes through. Reflects how real GPT-2 output looks.

### Fix 3: Lenient schema validator

`UnityCard._validate_text_complete` now only rejects empty / <30-char text. The truncation gate in script 13 already catches genuinely broken fragments — the schema doesn't need to duplicate that check.

### Fix 4: Faithfulness audit lexicon expanded

`INDICATOR_MENTIONS` in script 26 was too narrow — required exact words like "emotion" / "polariz" / "conspirac" — but the actual response-bank prose uses synonyms ("react", "tribal", "secret-cabal"). Expanded each indicator's lexicon to match the bank's actual word choices. Audit pass rate restored to 100%.

### Fix 5: Notebook cell 13 CLI corrected

Old notebook called `--in_jsonl/--target/--out_scored/--out_final` (legacy CLI). v2.1 script 13 accepts BOTH old and new CLI names via `argparse` aliases, so it works regardless.

## Smoke-test verification

50 realistic GPT-2 records (matching script 12's actual schema, 50% truncated):

```
Score → 50/50 passed (vs 6% in v2.0)
Theme → 50/50 accepted (vs ~6% in v2.0)
Curate → 12 promoted (3 REAL + 9 FAKE, all 3 candidates represented)
Faithfulness audit → 12/12 passed (100%)
Diversity → 12/12 unique summaries (100%)
```

All 38 unit tests pass.

## Accuracy preservation note

The detector-ensemble accuracy (RoBERTa + DistilBERT + DE-GNN + RandomForest + QLattice) is **unchanged** in v2.1. The fixes are entirely in the post-detection layer (verdict assignment, explanation, theme, pseudonymization, curation, audit). The trained models from previous studies are loaded and used identically; v2.1 just no longer drops their outputs by accident.
