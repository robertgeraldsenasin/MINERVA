# M.I.N.E.R.V.A. v2.6.final - Pipeline Integration

This document specifies the exact pipeline order for v2.6.final and shows
where each new component plugs in. Two modes are supported:

- **Mode A (templates):** rule-constrained scenarios from `scripts/30_template_scenario_generator.py`. Fast, deterministic, no GPU. Default for demos.
- **Mode B (GPT-2):** conditional generation via the paper's architecture (DE-GNN graph tokens + ensemble3 acceptance + QLattice scoring). Requires Colab GPU and JCBlaise download.

In both modes, **script 33 (strict allowlist enforcer) is the final filter** before the pool reaches Unity.

## How DE-GNN, QLattice, and detector features condition GPT-2

This is the answer to the user's question:
> "Shouldn't we be utilizing QLattice and the features extracted from the detectors and also using Dual Embedding Graph Neural Network to help generate using GPT-2?"

Yes. Per the paper, all three algorithms act as **teachers** to GPT-2, in three different ways:

### 1. DE-GNN signals condition the corpus before training (script 10)

Script 10 reads `data/features/degnn_preds.csv` (output of script 09) and converts each row's `p_degnn_fake` into a graph-confidence token:
- `p >= 0.80` -> `<|graph=high|>`
- `0.60 <= p < 0.80` -> `<|graph=mid|>`
- `p < 0.60` -> `<|graph=low|>`

Each training line becomes `<|label=...|> <|graph=...|> <text>`. GPT-2 fine-tunes on this conditioned corpus, learning a joint distribution `p(text | label, graph)`.

### 2. Detector features filter the generation in real time (script 12)

For each batch of generated candidates, script 12:
1. Computes `p_roberta_fake` via the trained RoBERTa-Tagalog detector
2. Computes `p_distil_fake` via the trained DistilBERT-multilingual detector
3. Encodes each candidate through both detectors' CLS, projects through PCA
4. Constructs the DE-GNN feature vector (lexical + p_rob + p_dis + r_pca + d_pca)
5. Pushes the feature vector through the trained DE-GNN GraphSAGE model -> `p_degnn_fake`
6. With `--accept_mode ensemble3`: averages all three probabilities, requires `>= min_conf` for the target label

This is a **three-vote acceptance gate**. A generated card has to convince all three independent algorithms it belongs to its target class. Cards that fail are dropped without ever leaving script 12.

### 3. QLattice scores accepted candidates (script 13)

Script 13 applies the symbolic regression equation from script 08 to each accepted candidate's feature vector. The equation produces:
- `p_fake_qlattice` (a calibrated fake-likelihood scalar)
- `margin = abs(p_fake_qlattice - 0.5)` (used as difficulty: large = easy/clear, small = hard/ambiguous)

The cards with the strongest margins make the best teaching material. The QLattice equation is **interpretable** (a small mathematical expression like `logreg(0.371*dpca1 + 0.339*rpca0 + 0.852*exp(0.533*dpca3) - 0.69)`), so panel questions about "why is this card scored 0.86 fake?" can be answered by tracing the equation.

### Why the GPT-2 base model's pretrained status is preserved

The paper's architecture does not erase GPT-2's pretrained Tagalog knowledge. Three reasons:

1. **The fine-tuning corpus is the JCBlaise text the base model already understands.** The only difference is the prepended conditioning tokens, which are new vocabulary additions, not replacements.
2. **The conditioning tokens are learned in their own embeddings.** GPT-2's existing word embeddings stay close to their pretrained values; the deltas concentrate in the new `<|label=...|>` and `<|graph=...|>` token embeddings.
3. **The ensemble3 gate + QLattice scoring filter the OUTPUT, not the model.** If GPT-2 ever produces something that drifts from the conditioning, it gets rejected at acceptance time. The base model is never punished for retaining pretrained behavior.

This is why no LoRA or adapter-based technique is needed: **the conditioning + filtering architecture already delivers the "retain pretrained, add task-specific" property** the user asked for.

## Defense talking points

When asked about each algorithm's role:

- **RoBERTa-Tagalog:** Tagalog-specialist detector. Trains on JCBlaise. Used as feature generator (CLS embeddings -> PCA) and as one of three voters in the ensemble3 acceptance gate.

- **DistilBERT-multilingual:** Multilingual generalist for code-switching. Same role as RoBERTa - feature generator + ensemble voter.

- **PCA over dual embeddings:** Reduces the 768-dim CLS embeddings from each detector to 12 dims each, making downstream tabular models (RF, QLattice, DE-GNN) feasible.

- **Random Forest:** Tabular baseline using the dual-PCA features. Provides feature-importance signals for debugging and confirms the neural detectors aren't doing something the simpler model can't match.

- **QLattice (Feyn):** Symbolic regression that produces a small mathematical expression mapping detector features to fake-likelihood. Versionable, interpretable, deterministic. Used to score every generated card.

- **DE-GNN (GraphSAGE):** Graph neural network over a kNN similarity graph of the dual-PCA feature space. Aggregates neighborhood signals to give a third independent vote on each card. Its `p_degnn_fake` output is what conditions GPT-2 generation via graph tokens AND what gates GPT-2 outputs via ensemble3.

- **GPT-2-Tagalog:** Causal language model fine-tuned on `<|label|> <|graph|> <text>` corpus. Generates conditioned text given a desired (label, graph) pair as prompt prefix.

- **Pseudonymizer (script 31):** Catches person-names in any source text and replaces with codes. Allowlist-protective.

- **Strict allowlist enforcer (script 33, NEW v2.6.final):** Final filter. Allowlist-enforcing - rejects any card whose text contains a name not on the codes-only allowlist (Candidate A/B/C). Catches whatever the pseudonymizer missed.

The story for the panel: M.I.N.E.R.V.A. is not "a model"; it is a **pipeline of complementary algorithms** where each one does one job correctly, and the whole system has independent layers that catch each other's mistakes.
