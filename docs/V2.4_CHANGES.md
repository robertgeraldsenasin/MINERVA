## What v2.4 fixes (the headline issue)

The thesis specifies exactly **three fictional candidates**: C-RM (Marquez), C-IB (Bantayan), C-JS (Salonga). Yet the v2.3 first-run audit found 125 of 200 cards (62.5%) being rejected because GPT-2 emitted placeholders like `Entity A`, `Entity B`, `Entity C` … `Entity V`, `Candidate W`, `Candidate AW` and they were never rewritten to the canonical three.

The bug was a regex gap in two places:

| File | v2.3 regex | What it caught | What it missed |
|---|---|---|---|
| `minerva_filters.py` | `Candidate\s+[A-Z]{3}\|Entity\s+[A-Z]\b\|Person\s+[A-Z0-9]{2,}` | 3-letter Candidates only; single-letter Entities only | Entity AA, Candidate W, Candidate AB, Person F |
| `22_pseudonymize_entities.py` | `[Cc]andidate\s+[A-Z]{1,3}` | Any Candidate code | Every Entity X and Person X |

Result: the *detection* regex caught Entity D's enough to flag those cards as "tainted", but the *rewrite* regex didn't catch them, so the cards entered the theme filter still tainted and got rejected wholesale.

**v2.4 unifies both regexes** to catch all three placeholder families with 1-3 letter codes:

```python
\b(?:Candidate|Entity|Person)\s+[A-Z]{1,3}\b
```

Now every placeholder GPT-2 emits gets rewritten to the cue-routed canonical candidate (C-RM, C-IB, or C-JS). No card is rejected for having a placeholder anymore.

## Detailed changes

### Patch P1 — Unified placeholder regex (the big one)

**Problem:** Two regex gaps. The detection regex caught 1-letter Entities and 3-letter Candidates only; the rewrite regex caught all-letter Candidates only. Mismatch → cards flagged as tainted but never repaired.

**Fix:** One unified regex in both `minerva_filters.py` and `22_pseudonymize_entities.py`:

```python
_LEGACY_PSEUDONYM_RE = re.compile(
    r"\b(?:Candidate|Entity|Person)\s+[A-Z]{1,3}\b"
)
```

This catches:
- `Entity A`, `Entity B`, …, `Entity V` (single letter — common in JCBlaise)
- `Candidate W`, `Candidate AW`, `Candidate EK` (1-3 letters — GPT-2 generations)
- `Person F`, `Person GG` (rare but observed)

**Why this is the right fix.** The thesis specifies exactly three candidates. Anything else GPT-2 produces is a placeholder that needs to be mapped onto one of the three. The cue router in `pseudonymize()` already chooses *which* of the three based on the card's narrative content (dynastic / reformist / populist cues). The regex just feeds it more inputs to route.

### Patch P2 — Alignment guard threshold

v2.2 introduced the verdict-rule alignment guard (REAL → UNCERTAIN when fired indicators ≥ 2). v2.3 audit showed it was too aggressive: only 4 REAL cards survived in the pool, vs. the 21 needed for the credible-card quota.

v2.4 raises threshold to ≥3 fired indicators. Reasoning:
- 2 indicators is common in any longer GPT-2 generation (MISS + FAB fire whenever there's a long quote and no URL — most posts).
- 3 indicators means the card has *meaningful* misinformation cue density, not just baseline noise.
- This restores REAL-card supply while keeping the protection against "REAL verdict + obviously misinformation cues" contradictions.

### Patch P3 — Proportional tier ratio

v2.0–v2.3 used absolute thresholds in `tier_for_card_index()`:
```python
if idx < 10: return "novice"
if idx < 25: return "proficient"
return "advanced"
```

This worked for the original 56-card decks. But when applied to a 200-card balanced pool (or even 64), >75% of cards have idx ≥ 25 and get tagged advanced.

v2.4 adds an optional `total_in_session` parameter. When provided, thresholds become proportional:
- 40% novice (idx < 0.40 × N)
- 35% proficient (0.40 ≤ idx < 0.75 × N)
- 25% advanced (idx ≥ 0.75 × N)

The 40/35/25 split reflects SHS pedagogy: front-load skill scaffolding, gradually transition to advanced critical-thinking prose. Caulfield & Wineburg (2023) advocate this progression for verification literacy.

### Patch P4 — REAL generation budget bump

`N_GENERATE_REAL` bumped from 1500 to 2500 in cell 1 of the notebook. REAL generations have empirically shown higher attrition in scoring (because real-news-shaped text is harder for GPT-2 to produce convincingly). Bumping the budget compensates.

This adds ~10 minutes to GPU runtime in cell 12 but ensures the pool has enough REAL supply after all the filtering.
