# M.I.N.E.R.V.A. v2.5 — Text Quality Patches

**Released:** 03 May 2026
**Status:** Tested end-to-end on v2.4 deliverable; six of seven text-quality issues resolved.
**Compatibility:** Drop-in replacement; no retraining required.

---

## Why this exists

The v2.4 deliverable's reports all looked green:
- 99.78% faithfulness audit
- 15.71% mean pairwise overlap (target <15%)
- 100% theme acceptance (0 legacy pseudonyms rejected)
- All three candidates routed correctly (282/91/72 verdict mix)

But the **actual card text was unreadable**. Random sampling of the 445-card pool revealed:

- **93.5% of cards truncated mid-sentence** (GPT-2 max_new_tokens=120 cuts off mid-thought)
- **40.9% had ≥5 surname mentions** in a single ~500-char card (severe name-jamming)
- **31.7% contained gibberish role codes** like "Salonga BS", "Marquez EA", "Bantayan BBE" — JCBlaise corpus artifacts hallucinated as random letter combos
- **4.7% had 4-8 letter placeholder leaks** ("Candidate FIVB", "Entity HDERYL") that v2.4's 1-3 letter regex cap missed
- **2.7% had pure-English quote chunks** (15+ words, zero Tagalog markers) inserted into Tagalog narrative
- **2.0% mentioned real Filipino political names** (Lopez, Panelo, Radaza, Paolo) not in the v2.4 surname allowlist

The pipeline metrics were measuring whether explanations matched indicators (faithfulness) and whether decks were diverse (overlap), but **never measured whether the cards were readable**. v2.5 closes that gap.

## What v2.5 changes

Seven patches in four files, all post-processing — no retraining needed.

### Files changed

| File | Patches | Purpose |
|---|---|---|
| `scripts/22_pseudonymize_entities.py` | P1, P2, P3, P3b, P4, P7 | New text-cleanup helpers + extended regex |

| `scripts/minerva_candidates.py` | P5 | Extend surname allowlist |
| `scripts/13_score_generated_with_qlattice.py` | P6 | Drop unsalvageable cards at score stage |

### Patch details

**P1 — Extended placeholder regex (1-3 → 1-8 letters)**
Both `minerva_filters.py` and `22_pseudonymize_entities.py` change from `[A-Z]{1,3}` to `[A-Z]{1,8}`. Catches the 21 cards with codes like FIVB, IZZM, DDDU, BSEK, JUWA, BREA, HDERYL that v2.4 missed.

**P2 — Gibberish suffix-code cleanup** (`_strip_gibberish_codes()`)
For each canonical surname (Marquez/Bantayan/Salonga), strips any trailing 1-5 letter ALL-CAPS token (BS, EA, BBE, DUBA), unless it's a legitimate honorific (Jr, Sr, II, III, IV). Also strips parenthesized variants like "Marquez (BS)" and "Marquez (Marquez)". Affected 31.7% of v2.4 deliverable.

**P3 — Mid-sentence truncation trim** (`_trim_trailing_fragment()`)
Strategy:
1. If text ends in `.!?"` already, return as-is.
2. Find the last terminal punctuation in last 250 chars; if found and content before is ≥50 chars, trim there.
3. Otherwise split on sentence boundaries and drop the last (truncated) sentence if remaining is ≥50 chars.
4. Last resort: append `...` to mark truncation honestly.

Affected 93.5% of v2.4 deliverable cards (the biggest single fix).

**P3b — Mid-word cleanup** (inline)
After truncation trim, the last token is checked. If it's a 1-2 char fragment that isn't a legitimate Tagalog particle (`sa`, `ng`, `ay`, `si`, `ni`, `po`, `rin`, `din`, `na`, `ko`, `mo`), it gets dropped.

**P4 — Strip incoherent English-only quotes** (`_strip_incoherent_english_quotes()`)
Detects double-quoted blocks of ≥60 chars / ≥15 words containing zero common Tagalog function words (ang, ng, sa, ay, mga, kay, etc.). Strips them. The fine-tuned GPT-2 occasionally inserts random English narrative chunks into Tagalog cards; this removes them while preserving short quotes and Tagalog-content quotes.

**P5 — Extended `_SURNAME_ALLOWLIST`**
Adds 28 names observed as residual leaks: Lopez, Panelo, Radaza, Paolo, Andanar, Roque, Bong, Bongbong, Imee, Inday, Sara, Garin, Lapid, Belmonte, Mangudadatu, Razon, Romualdez, Ejercito, Dela Rosa, Bato, Tugade, Lorenzana, Galvez, Año, Esperon, Magalong, Dichaves, Ressa. **Does NOT** include canonical surnames (Marquez, Bantayan, Salonga) — those are the rewrite targets, not leaks.

**P6 — Score-stage drop policy**
Extended `should_drop()` to reject cards that are unsalvageable at the source:
- `gibberish_code_dense_N` — text contains ≥4 ALL-CAPS gibberish-code suffixes after canonical surnames
- `name_jammed_<surname>` — a single canonical surname appears ≥8 times

In a fresh pipeline run, these 41 cards (9.2% of the v2.4 deliverable) get dropped before scoring instead of being polished post-hoc.

**P7 — Adjacent surname collapse**
After P2 strips "Salonga BS, Salonga DUBA, Salonga BH" → it becomes "Salonga, Salonga, Salonga" — same surname count, just no codes. P7 collapses adjacent or near-adjacent repeats: "X, X, X" → "X", "X at X" → "X", "X ni X" → "X". Recovered 56 additional cards from name-jamming.

## Limitations

**Structural name jamming (28% residual).** The fine-tuned GPT-2 repeats the candidate's surname every sentence as the natural anaphoric reference. P7 collapses adjacent repeats but cannot rewrite "Marquez na sinabi ni Marquez tungkol kay Marquez" into idiomatic Tagalog. The 28% residual is awkward but readable. Eliminating it would require either retraining the generator on cleaner data or running an LLM rewrite pass on each card.

**The 2.0% residual truncation.** Nine cards end with `...` because they were too short or fragmented for `_trim_trailing_fragment()` to find a 50-char-minimum cut point. These are honest excerpts now rather than dishonest mid-word cutoffs.

**Indicator coverage at 4 of 12.** Only MISS, FAB, ANON, URG ever fire. The other 8 (EMO, POL, CONS, DISC, IMP, REV, ENDO, RECF) require GPT-2 prompt scaffolding. Deferred — same as in v2.4.

## How to apply v2.5

Two modes:

### Mode A — Apply to existing v2.4 deliverable (no GPT-2 regen, ~5 min)

If you already have v2.4 outputs in `generated/`, run the v2.5 patches via re-running script 22 onwards:

```python
# Cell 16 in your notebook (re-run with the new script 22)
!python scripts/22_pseudonymize_entities.py \
    --in_file  generated/unity_cards_balanced.json \
    --out_file generated/unity_cards_pseudonymized.json \
    --re_explain --seed 1729

# Cells 17-22 unchanged — they pick up the cleaned text
```

This applies P1-P5 + P7 to the existing data. P6 only matters for fresh runs.

### Mode B — Full fresh pipeline (~30 min GPU + ~5 min)

For the panel-defense-grade run, regenerate from scratch with the v2.5 score-stage drop policy in effect. Use the v2.4 notebook (it's already configured for the 500-card pool).

The fresh run will also:
- Drop 9.2% of GPT-2 generations at score stage as unsalvageable (P6)
- Apply all post-processing patches downstream
- Yield a cleaner pool with a higher % of readable cards (estimated 95-97% readable based on v2.5 audit)
