### P1 #3 · "Get faithfulness back to ≥98%"

**Audit finding:** Faithfulness regressed from 100% (v2.8.5) to 87.18% (v2.8.7). Root cause: `scripts/29_merge_gpt2_into_pool.py::_build_explanation` emitted stub strings ("GPT-2 generation flagged X") that didn't match the actual indicator phrasing the audit script expects, producing 231 indicator-phrase mismatches.

**v2.9.0 response (two-pronged fix):**

1. **`_build_explanation()` rewritten** to consume the v2.9 response bank instead of stubs. Each fired indicator gets a real Tagalog phrase from `templates/response_bank_v2.json` keyed by `{indicator}/{role}/{tier}`. For real-verdict cards, also auto-appends a `CREDIBLE/real/{tier}` phrase.

2. **NEW pre-merge filter** `gpt2_indicators_supported_by_bank()` runs *before* schema mapping. If any of a GPT-2 card's fired indicators don't have a response-bank entry for its (role, tier), the card is rejected with `stage: "response_bank_coverage"`. This guarantees no card with mismatched-phrase risk reaches the merged file. Two layers of defense.

**Verification:** `reports/faith.json::pass_rate` should be ≥98% after re-run (was 87.18%). `reports/merge_gpt2_into_pool.json::gpt2_rejected_by_reason` should show a new `response_bank_coverage` category.

**Files touched:**
- UPDATED: `scripts/29_merge_gpt2_into_pool.py` (`_build_explanation` rewritten, filter added in merge loop, ~100 new lines)
- NEW: `templates/response_bank_v2.json` (225 phrase entries, 75 keys, 72% diversity)
