# M.I.N.E.R.V.A. v2.3 — Dynamic Content System

**Released:** 03 May 2026
**Status:** Tested end-to-end on real data; pool curation, per-user draw, and faithfulness audit all green.
**Compatibility:** Drop-in replacement; no retraining required (existing detector models still apply).

---

## What v2.3 changes

The previous releases (v2.0 → v2.2) all produced one fixed 56-card deck per Colab run. Every Filipino SHS student who installed the resulting Unity APK would see the **same 56 cards in the same order**. That's a teaching curriculum, not dynamic content.

Two real problems with that approach:
1. **Educational integrity collapses** — students could share screenshots/answers; the second student to play already knows the answers from the first.
2. **No statistical power for the thesis evaluation** — if two students' decisions are on identical cards, those decisions are not independent observations.

v2.3 fixes both by separating **pool curation** from **per-user deck draw**.

---

## The new architecture

```
Before (v2.0–v2.2):
  GPT-2 (1000)  →  scored (~800)  →  balanced (200)  →  themed (180)
                                                          ↓
                                          curated single 56-card deck
                                                          ↓
                                                  Unity loads as-is

After (v2.3):
  GPT-2 (3000)  →  scored (~2400)  →  balanced (500)  →  themed (~450)
                                                          ↓
                                                  POOL of 500 cards
                                                          ↓
                                          script 28: per-user draw
                                                          ↓
                                            56 cards per user_id
                                                          ↓
                                  Unity loads pool + draws at runtime
```

## Detailed changes

### 1. New script: `28_draw_user_deck.py`

The heart of v2.3. Given a user_id and the pool, deterministically samples a 56-card deck respecting all quotas:

- **REAL quota:** ≥3 REAL cards per day (Modirrousta-Galian & Higham 2023 mandate)
- **Candidate-coverage quota:** at least 1 card per candidate per day when pool supports it
- **Mixed verdict per day:** ~85% FAKE, ~15% UNCERTAIN, REAL quota first
- **Per-day shuffle:** REAL cards aren't all clustered at the start of each day
- **Cross-linking:** each FAKE card links back to the most recent REAL the player has seen (for VERIdict's credible-counter pairing)

**Determinism.** Uses SHA-256 of `f"{user_id}:{pool_hash}"` as the RNG seed. Easy to port to C# (BitConverter.ToInt64 of the first 8 bytes of the SHA-256 hash).

**Output structure:**
```json
{
  "_metadata": {
    "user_id": "alice",
    "pool_hash": "8767f4e77f9769a4",
    "draw_seed": 1617633261007446191,
    "days": 7,
    "cards_per_day": 8,
    "min_credible_per_day": 3,
    "deck_size": 56,
    "verdict_dist": {"REAL": 21, "FAKE": 30, "UNCERTAIN": 5},
    "candidate_dist": {"C-IB": 18, "C-JS": 24, "C-RM": 14},
    "indicator_coverage": {"MISS": 54, "FAB": 20, "ANON": 11, "URG": 8}
  },
  "cards": [ ... 56 story-card objects ... ]
}
```

### 2. Rewritten: `24_curate_teaching_cards.py`

- **CLI changed:** `--target_pool_size` (default 500) replaces `--target_total`
- **Output:** `unity_cards_pool.json` instead of `story_cards.json`
- **No day assignment:** cards are pool-indexed; days are assigned at draw time
- **Output format:** `{"_metadata": {...}, "cards": [...]}` instead of flat list
- **Backward compat:** old `--days/--cards_per_day/--min_credible_per_day` flags accepted but informational only (carried into pool metadata for the draw script to read)

### 3. Updated: `21_balance_unity_cards.py`

- **Default `--target_total` bumped from 200 → 500**

This is the upstream supply that feeds the pool. With 500 cards balanced across verdict × candidate × indicator dimensions, the downstream curator has enough variety to produce a 500-card pool that supports diverse per-user decks.

### 4. Schema: `minerva_schemas.py`

- `StoryCard.day` made optional (`int | None`, default None)
- `StoryCard.pool_index` added (set by curator, helps Unity track pool position)

Pool cards have `day=None`. Drawn deck cards have `day=1..7`. Unity's loader can detect which by checking the field.

### 5. Loader helper in scripts 25, 26, 27

```python
def _load_cards_or_pool(path: str) -> list:
    payload = json.load(open(path, encoding="utf-8"))
    if isinstance(payload, dict) and "cards" in payload:
        return payload["cards"]
    return payload
```

This makes the downstream scripts (VERIdex builder, faithfulness audit, bank-version stamper) work with both old flat-list inputs and new pool-dict inputs without breaking changes.

## Pool sizing: how many cards do you need?

The math:

```
Cards per player        = DAYS_IN_DECK × CARDS_PER_DAY = 7 × 8 = 56
Expected pairwise overlap = cards_per_player / pool_size
```

| Pool size | Max non-overlap decks | Pairwise overlap | Use case |
|---|---|---|---|
| 200 | 3 | ~28% | Cohort ≤ 5 students (high overlap risk) |
| **500 (v2.3 default)** | **8–9** | **~11%** | Cohort ≤ 50 students with low overlap |
| 800 | 14 | ~7% | Cohort ≤ 100 students |
| 1000 | 17 | ~6% | Cohort ≤ 150 students |

For the **first SHS pilot (≤50 students), 500 cards is recommended.**
For the **full thesis evaluation (~100 students), bump pool to 800-1000** by setting `POOL_SIZE = 800` in cell 1.

## Unity team integration guide

The Unity team has two options for shipping the dynamic content:

### Option A — Offline draw (recommended for current scope)

1. Ship `unity_cards_pool.json` with the APK (single file, ~2 MB).
2. Port `28_draw_user_deck.py` logic to C#:
   - SHA-256 hash of `user_id + ":" + pool_hash` (use `System.Security.Cryptography.SHA256`)
   - Take first 8 bytes, interpret as Int64 with `BitConverter.ToInt64`
   - Use as seed for `System.Random` (or any deterministic PRNG)
   - Replicate the per-day draw quotas from the Python script
3. On first launch, the user enters their student ID. Unity computes their deck. Same student ID = same deck. Different student ID = different deck.

### Option B — Server-side pre-built decks

1. Run `28_draw_user_deck.py` server-side for each enrolled student ID, producing N deck files.
2. Ship deck files via a backend API. Unity downloads `deck_<user_id>.json` on first launch.
3. Pros: no C# port needed. Cons: requires backend, breaks the "no live network" scope.

**Recommendation: Option A.** It satisfies §1.5's "no live social media streams" delimitation and keeps deployment simple.

## How to run v2.3 (no retraining needed)

You already have v2.1 detector models trained. v2.3 only changes the post-detection scripts (24, 28) plus minor loader updates to 25/26/27.

1. Pull patches:
   ```powershell
   cd D:\MINERVA\MINERVA
   git pull origin upgrade/minerva-election-theme
   ```

2. **Re-generate GPT-2 samples with the bumped budget** (1500 each instead of 500). Cell 12 in the Colab notebook does this automatically — just re-run from cell 12 onwards. **GPU time: ~30-45 min.**

3. **OR skip regeneration if you're OK with a smaller pool.** Just re-run cells 14–22 with whatever scored samples you already have. Pool will be smaller (~150-200 cards instead of 500), capacity for unique decks lower (~3 instead of 9).

4. **Verify metrics** at cell 22:
   - `pool_curation_report.json` should show ~500 cards
   - `draw_report.json` should show pairwise overlap ~11% (or higher if pool is small)
   - `faithfulness_audit_report.json` should show ≥95% pass rate

## What is NOT in v2.3 (deferred)

- **Pool versioning.** If you regenerate the pool mid-study, students with already-drawn decks need a migration path. v2.3 stamps `pool_hash` on each deck; v2.4 will add explicit pool_version + drift handling.
- **Adaptive difficulty per user.** Currently the per-day mix is fixed (35% REAL, 60% FAKE, 5% UNCERTAIN). v2.4 could adapt difficulty based on the user's past accuracy.
- **C# reference port of script 28.** Not delivered in v2.3. The Python script is the reference; Unity team writes the C# port.

## Acknowledgement of imperfections

- **Pool size depends on upstream supply.** If you don't bump `N_GENERATE_FAKE` and `N_GENERATE_REAL` to 1500 each (cell 1 default in v2.3), you'll end up with a pool of ~200 instead of 500.
- **Pool exhaustion is logged but not failed.** If a player draws and the per-day quotas can't be met (e.g., not enough REAL cards left), the script logs a warning and gives the player a smaller day. The thesis evaluation should plan around this — for ≤50 students with a 500-card pool, exhaustion is not a real risk.
- **The C# port for Unity is not yet written.** The Python reference is correct; porting requires ~1 day of Unity-team effort.
